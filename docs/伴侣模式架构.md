# 伴侣模式 (Companion Mode) 架构文档

## 一、项目整体结构

```
ai_virtual_mate_comm/
├── partner/                          # 伴侣模式核心模块
│   ├── characterStatus.py            # 角色状态管理类 (数据封装 + 持久化)
│   ├── companion.py                  # 对话桥接模块 (prompt构建 → LLM调用 → 状态解析)
│   ├── actions.py                    # 行为定义 (6 种主动行为 + 装饰器注册)
│   ├── actionScheduler.py            # 行为调度器 (模式入口)
│   └── config.json                   # 伴侣配置文件 (状态存储)
│
├── llm.py                            # LLM 模块 (多后端路由, 伴侣模式委托 companion.py)
├── ase.py                            # 主动感知模块 (触发伴侣行为)
├── sys_init.py                       # 系统初始化 (加载伴侣配置)
├── main.py                           # 主程序入口
├── main_sub.py                       # 通用对话处理
├── chat_web.py                       # Web 聊天界面
├── gui.py                            # GUI 主界面
└── data/db/
    ├── config.json                   # 主配置文件
    ├── preference.json               # 用户偏好设置
    └── memory.db                     # 对话历史记忆
```

## 二、核心组件详解

### 2.1 characterStatus.py — 角色状态管理

**职责**: 封装和管理角色的所有状态信息，实现内存与配置文件的同步。

**设计模式**: 单例 (全局 `partner_config` 对象)

**状态属性**:

| 属性 | 说明 | 类型 |
|------|------|------|
| `_current_location` | 角色当前位置 | string |
| `_current_time_period` | 时间段 (早上/中午/下午/晚上/凌晨) | string |
| `_current_action` | 角色当前行为 | string |
| `_agreed_events` | 与用户的约定事件队列 (FIFO) | list |
| `_current_location_options` | 可选位置列表 (自迭代增长) | list |
| `_current_action_options` | 可选行为列表 (自迭代增长) | list |
| `_is_user_nearby` | 用户是否在角色身边 | bool |
| `_choice_next_action` | LLM 选择的下一步行为 | dict |

**核心方法**:

```
Getter 方法:
  get_current_location()        → 获取位置
  get_current_time_period()     → 获取时间段
  get_current_action()          → 获取行为
  get_agreed_events()           → 获取约定事件列表 (返回副本, 防外部篡改)
  get_current_location_options()→ 获取可选位置
  get_current_action_options()  → 获取可选行为
  get_is_user_nearby()          → 获取用户是否 nearby
  get_choice_next_action()      → 获取下一步行为配置 {"action": ..., "params": ...}

Setter 方法 (均通过 _save() 同步写入 config.json):
  _save()                       → 内部方法, 统一将所有状态写入文件
  set_current_location(value)
  set_current_time_period(value)
  set_current_action(value)
  set_is_user_nearby(value)
  put_agreed_event(value)         → 添加约定事件 (入队)
  take_agreed_event()             → 取出并移除第一个约定 (出队)
  set_choice_next_action(action, params)
```

**持久化设计**:

所有 setter 调用统一的 `_save()` 方法将全部状态一次性写入文件, 避免每个 setter 重复写文件逻辑。`_save()` 将内存中的 `_current_location`、`_current_time_period`、`_current_action`、`_agreed_events`、`_current_location_options`、`_current_action_options`、`_is_user_nearby`、`_choice_next_action` 同步到 `config_json['last_status']` 后一次性写入。

### 2.2 companion.py — 对话桥接模块

**职责**: 伴侣模式专用的对话入口, 负责构建 prompt、调用 LLM、解析 JSON 响应并更新状态。

**核心函数**:

```
build_companion_prompt(base_prompt, partner_config)
  → 构建伴侣模式专属系统提示词 (含角色状态 + JSON 格式约束)

update_companion_state(partner_config, res_json, openai_history, think_filter_switch)
  → 解析 LLM 返回的 JSON, 更新角色状态、约定事件、下一步行为

companion_chat(msg, openai_history, client, custom_model, partner_config, think_filter_switch, base_prompt)
  → 完整对话入口: 构建 prompt → 调用 LLM → 解析 JSON → 更新状态 → 返回回复
  → 失败时回退到原始文本返回, 保证系统可用
```

**设计意图**: 将伴侣模式特有的对话逻辑从 `llm.py` 的超长 if-elif 链中抽离, 实现关注点分离 (SoC)。`llm.py` 只需一行调用即可完成委托:

```python
return companion_chat(msg, openai_history, client, custom_model, partner_config, think_filter_switch, prompt)
```

### 2.3 actions.py — 行为定义

定义了 6 种伴侣可执行的主动行为:

| 行为 | 函数 | 触发场景 |
|------|------|----------|
| **自言自语** | `action_self_talking()` | 角色独处、无明确事项 |
| **推动约定事件** | `action_push_agreed_event()` | 有未完成约定时优先 |
| **与他人对话** | `action_talk_with_other(name=None)` | 需主动找其他人说话 |
| **表达身体状态** | `action_express_body_state()` | 有明显身体状态 (饿/累/困) |
| **与环境互动** | `action_interact_with_environment()` | 当前环境有可互动元素 |
| **默认行为** | `action_default()` | 不符合以上任何情境 |

**行为执行模板**:

```python
def _run_action(msg, notice_text, stream_prefix):
    """统一的行为执行流程"""
    stop_tts()              → 停止当前语音
    chat_llm(msg)           → 调用 LLM 生成内容
    notice(notice_text)     → 发送通知
    stream_insert(...)      → 插入到聊天流
    get_tts_play(response)  → 语音合成并播放
    return response
```

每个 action 函数只需 3 行, 调用 `_run_action()` 并传入各自的 `msg`、`notice_text`、`stream_prefix` 即可。

**装饰器注册** (开闭原则):

```python
registered_actions = {}  # 由装饰器自动填充

def register_action(name):
    """行为注册装饰器"""
    def decorator(func):
        registered_actions[name] = func
        return func
    return decorator

@register_action('action_self_talking')
def action_self_talking():
    return _run_action(msg="...", notice_text="...", stream_prefix="...")
```

添加新行为只需写函数 + 加装饰器, 无需修改 `actionScheduler.py`。

### 2.4 actionScheduler.py — 行为调度器

**职责**: 管理行为调用计数和触发逻辑, 是伴侣模式主动行为的调度入口。

**调度策略**:

```
run_action() 流程:

  other_actions_count = action_counts['action_self_talking']
                      + action_counts['action_talk_with_other']

  if other_actions_count >= 2:
      # 防沉迷: 闲聊过多时强制推进约定事件
      reset_counts()
      → 执行 action_push_agreed_event

  else:
      # 随机选择行为执行
      # (无约定事件时排除 action_push_agreed_event)
      → random.choice(registered_actions)
      → action_counts[action_name] += 1
```

**行为注册**: `registered_actions` 从 `partner.actions` 模块导入, 由 `@register_action` 装饰器自动填充, 无需手动维护注册表。

### 2.5 config.json — 伴侣配置

**关键结构**:

```json
{
  "response_rule": "...",
  "response_rule_schema": { /* JSON Schema 定义 */ },
  "response_rule_object": {"type": "json_object"},
  "last_status": {
    "location": "月海亭",
    "time_period": "中午",
    "action": "去吃饭",
    "agreed_events": [...],
    "is_user_nearby": true,
    "current_location_options": [...],
    "current_action_options": [...],
    "choice_next_action": {
      "action": "action_push_agreed_event",
      "params": null
    }
  }
}
```

**LLM 响应格式约束** (`response_rule`):

LLM 必须返回如下 JSON:

```json
{
  "time_period": "早上/中午/下午/晚上/凌晨",
  "is_user_nearby": true/false,
  "action": "行为描述",
  "location": "位置",
  "is_completion": true/false,
  "is_new_event": true/false,
  "new_event": "新约定内容",
  "message": "回复内容",
  "choice_next_action": {
    "action": "action_self_talking / action_push_agreed_event / ...",
    "params": {"character_name": "..."} 或 null,
    "reason": "选择理由"
  }
}
```

通过 `data/db/preference.json` 中的 `运行模式切换` 字段切换:

| 模式 | 入口 | 特点 |
|------|------|------|

| 多智能体助手 | agent.py | 任务识别 + 智能体调用 |
| **伴侣模式** | partner/ | **LLM 驱动状态 + 行为调度 + 约定系统** |

## 五、核心设计思想

### 5.1 状态持久化
所有角色状态存储在 `partner/config.json`, 每次修改通过 `_save()` 同步写入文件, 保证重启后状态不丢失。

### 5.2 LLM 驱动决策
由 LLM 决定:
- 角色位置和行为变化
- 用户是否 nearby
- 时间段流转
- 约定事件的完成和新增
- 下一步主动行为选择 (`choice_next_action`)

### 5.3 双层调度策略
- **第一层**: 优先执行 LLM 在上一轮对话中选择的 `choice_next_action`
- **第二层**: 降级到 `run_action()` 随机调度, 并有防沉迷机制 (闲聊 ≥2 次强制推进约定)

### 5.4 约定事件系统
- FIFO 队列管理
- LLM 通过 `is_completion` 标记完成并移除
- LLM 通过 `is_new_event` 标记新增约定
- 推动约定事件是最高优先级的主动行为

### 5.5 自迭代选项
`current_location_options` 和 `current_action_options` 随使用自动增长, 新位置/行为自动添加到可选列表中。

## 六、关键数据流总结

```
用户输入 ──→ chat_llm() ──→ companion.py ──→ LLM (返回 JSON)
                                      │
                                      ▼
                            解析并更新 partner_config
                                      │
                      ┌───────────────┘
                      │
定时触发 ←─ ase.py ←─ choice_next_action
                      │
            run_action() (降级)
                      │
            执行具体行为 ──→ TTS 播放
```
